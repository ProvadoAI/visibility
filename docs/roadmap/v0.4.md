# v0.4 Roadmap

## Roadmap intent

v0.4 builds on the completed v0.3 deterministic one-product analyzer.

v0.1 delivered the deterministic core engine. v0.2 made the diagnostics clearer (taxonomy cleanup, visibility health, explicit URL evidence, focused fixtures, scenario CLI). v0.3 made reports more usable and scenarios more realistic (transport/indexability/schema/content scenarios, compact output, scenario validation, merchant-facing summary fields).

v0.4 takes the **first deliberate step across the external-data boundary** that the first three versions intentionally held.

The theme for v0.4 is:

> Real page-evidence acquisition, behind existing interfaces, without sacrificing deterministic testing.

This is the architecturally-indicated next step. `docs/architecture.md` already names `Psr18PageFetcher` as the "Later implementation" behind the `PageFetcher` interface and states that "HTTP fetching can be added behind an interface after fixture tests are stable." Fixture tests are stable. This version implements that seam.

### Why page fetching and not live search first

v0.4 makes the analyzer able to fetch the **expected product page** over real HTTP, read `robots.txt`, and inspect the sitemap — the "diagnostics" half of the product vision. It does **not** add live search-result acquisition.

Page fetching is the smaller, safer boundary crossing:

- it sits behind the existing `PageFetcher` interface, so detectors, the analyzer, and report shape are unchanged;
- search-result evidence is still supplied from outside the core, preserving "no acquisition coupling";
- it stays deterministically testable with a mock PSR-18 client, so the core test suite needs no network;
- the CLI keeps fixtures as the default; live fetching is strictly opt-in.

Live search acquisition (API keys, quotas, rate limits, non-determinism) is a larger jump that is better built **on top of** the HTTP plumbing and safety controls introduced here. It is recorded in `docs/deferred.md` and is a candidate for v0.5+.

## v0.4 product boundary

The v0.4 core remains centered on this API shape:

```php
VisibilityAnalyzer::analyze(
    ProductSubject $product,
    array $queries
): VisibilityReport;
```

Where:

- `$product` is one product.
- `$queries` contains one or more query contexts for that product.
- Search-result evidence is still supplied from outside the core through a provider or scenario fixture.
- Page evidence may now be supplied through a fixture fetcher **or** acquired live through a PSR-18 HTTP fetcher, both behind the same `PageFetcher` interface.
- The analyzer returns one `VisibilityReport` for that product.

There is no `BatchVisibilityAnalyzer` in v0.4.

The CLI remains scenario-driven and deterministic by default:

```bash
bin/visibility analyze examples/scenarios/visible-clean.json
```

Live page fetching is opt-in only:

```bash
bin/visibility analyze examples/scenarios/live-fetch.json --live
```

## v0.4 design principles

- Keep the analyzer deterministic. Live behavior is opt-in and lives behind interfaces.
- Keep one-product analysis as the central unit.
- The core depends on PSR HTTP **abstractions only**. It must not hard-depend on Guzzle, Symfony HttpClient, or any concrete client. The caller injects the implementation.
- Preserve "no acquisition coupling" for search: the core still does not collect search results.
- Preserve v0.2/v0.3 semantics: `visible` / `not_visible` / `uncertain`, `healthy` / `at_risk` / `blocked` / `unknown`, and the URL evidence roles `matchedUrl` / `requestedUrl` / `finalUrl` / `expectedUrl` / `canonicalUrl`.
- Preserve existing JSON report compatibility unless a change is explicitly documented and tested.
- Network access must be safe by default: bounded timeouts, bounded redirects, bounded body size, and explicit opt-in.

## v0.4 non-goals

Do not implement these in v0.4. See `docs/deferred.md` for the full deferred backlog and the conditions that would unlock each item.

- live Google scraping;
- live Bing scraping;
- live search-result acquisition of any kind (search evidence stays supplied);
- marketplace scraping;
- Google Shopping scraping;
- AI answer-engine checks;
- embeddings, vector databases, semantic product-query matching;
- automatic expected-query generation;
- browser rendering / JavaScript execution;
- full-site crawling or link-following beyond the expected product URL, its redirects, `robots.txt`, and the sitemap;
- Laravel integration;
- dashboard UI;
- SaaS account/project model;
- scheduled monitoring;
- competitor reports;
- batch analysis for many products;
- revenue forecasting or attribution modeling.

## Phase 1 — PSR-18 HTTP page fetcher

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Implement a real HTTP `PageFetcher` behind the existing interface so the analyzer can fetch the expected product page from the live web, while keeping the test suite deterministic.

### Problem

v0.1–v0.3 only fetch page evidence from fixtures. `docs/architecture.md` defines `Psr18PageFetcher` as the planned later implementation, but it does not exist yet, so the analyzer cannot be pointed at a real URL.

### Deliverables

- `VisibilityDetector\Adapters\Http\Psr18PageFetcher` implementing `PageFetcher`.
- Composer dependencies on PSR HTTP **abstractions only**: `psr/http-client`, `psr/http-factory`, `psr/http-message`.
- Constructor injection of a PSR-18 `ClientInterface` and PSR-17 request/URI factories, so any concrete client (Guzzle, Symfony HttpClient, etc.) can be supplied by the caller.
- Mapping of transport outcomes onto the existing `PageSnapshot` fields: `requestedUrl`, `finalUrl`, `statusCode`, `headers`, `body`, `contentType`, `redirects`, `durationMs`, `failureType`, `warnings`.
- Mapping of PSR-18 exceptions onto the existing `failureType` taxonomy: `dns_not_found`, `timeout`, `connection_refused`, `ssl_error`, `http_error`, `invalid_response`, `unknown`.

### Acceptance criteria

- `Psr18PageFetcher::fetch(string $url): PageSnapshot` returns a populated snapshot for a successful response.
- Non-2xx responses still return a snapshot with the correct `statusCode` (the fetcher captures evidence; the `HttpAvailabilityDetector` interprets it).
- Network-level failures are mapped to the correct `failureType` and never throw out of `fetch()`.
- Redirect chains are captured in `redirects`, with `finalUrl` set to the last URL.
- The core depends only on PSR interfaces; no concrete HTTP client is required by `composer.json` `require`.
- All tests use a mock/in-memory PSR-18 client. No test performs a real network call.
- `vendor/bin/phpunit` passes after implementation.

## Phase 2 — Fetch policy and safety controls

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Make live fetching safe and predictable by default with explicit, deterministic policy controls.

### Problem

Real HTTP introduces risks the fixture fetcher never had: hung connections, redirect loops, huge responses, and accidentally requesting hosts the operator did not intend. These must be bounded before live fetching is exposed through the CLI.

### Deliverables

- A `FetchPolicy` value object (or equivalent config) consumed by `Psr18PageFetcher`, covering:
  - connect/total timeout;
  - maximum redirects to follow;
  - maximum response body size;
  - a configurable User-Agent string;
  - an optional host allowlist and/or denylist.
- Deterministic enforcement: exceeding a bound produces a controlled `PageSnapshot` (appropriate `failureType` and/or `warnings`), not an uncaught exception.
- Safe defaults documented in code and README.

### Acceptance criteria

- A request exceeding the timeout yields `failureType = timeout`.
- A redirect chain longer than the configured maximum is stopped and recorded (warning and/or `failureType`), not followed indefinitely.
- A response body larger than the configured maximum is truncated or rejected deterministically with a recorded warning.
- A host that fails the allowlist/denylist check is refused before any request is made, with a clear warning and no network call.
- Policy is deterministic and unit-tested with a mock client.
- `vendor/bin/phpunit` passes after implementation.

## Phase 3 — robots.txt evidence

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Fetch and interpret `robots.txt` so the analyzer can detect when the expected product URL is disallowed from crawling — a real indexability blocker the v0.1 design already left a seam for.

### Problem

v0.1's `IndexabilityDetector` explicitly "should leave room for robots evidence," but live `robots.txt` fetching was deferred. With a real fetcher and safety policy in place, this evidence can now be acquired.

### Deliverables

- A `RobotsTxtFetcher` (using the Phase 1 fetcher + Phase 2 policy) and a deterministic `RobotsTxtParser`.
- A robots evidence structure indicating whether the expected URL is allowed/disallowed for a given User-Agent, plus the matched rule.
- Wiring so the existing `IndexabilityDetector` can emit a robots-blocked finding (reuse the existing `page.*` indexability namespace; add a single new code such as `page.robots_disallowed` only if no existing code fits, and document the choice).

### Acceptance criteria

- A `Disallow` rule matching the expected URL path produces a robots-blocked finding.
- An `Allow`/no-match case produces no robots-blocked finding.
- User-Agent-specific groups are honored deterministically.
- Missing or unfetchable `robots.txt` is treated as "not blocked" with a warning, never a fatal error.
- The parser is unit-tested independently of the network with raw `robots.txt` fixtures; fetch wiring is tested with a mock client.
- `vendor/bin/phpunit` passes after implementation.

## Phase 4 — Sitemap presence evidence

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Inspect the site's XML sitemap to detect whether the expected product URL is listed — a discoverability signal distinct from page-level indexability.

### Problem

A product page can be indexable and well-formed yet absent from the sitemap, which weakens discovery. v0.1 listed "sitemap and link evidence ideas" as a research influence but deferred implementation. This phase adds the smallest useful version: presence/absence of the expected URL.

### Deliverables

- A `SitemapFetcher` (Phase 1 + Phase 2) and a deterministic `SitemapParser` supporting `<urlset>` and `<sitemapindex>` (bounded, one level of index expansion).
- A new detector (e.g. `SitemapPresenceDetector`) with finding codes such as:
  - `sitemap.url_listed` (info);
  - `sitemap.url_missing` (low/medium);
  - `sitemap.unavailable` (info/low).
- Bounded parsing consistent with v0.4 non-goals (no full-site crawl): cap total URLs parsed and child sitemaps expanded.

### Acceptance criteria

- Expected URL present in the sitemap (matched via existing URL normalization) produces `sitemap.url_listed`.
- Expected URL absent from a fetched, parseable sitemap produces `sitemap.url_missing`.
- Missing or unparseable sitemap produces `sitemap.unavailable` with a warning, never a fatal error.
- Sitemap-index files expand one level, bounded by the configured caps.
- Parser is unit-tested with raw XML fixtures; fetch wiring tested with a mock client.
- This phase does not follow page links or crawl the site.
- `vendor/bin/phpunit` passes after implementation.

## Phase 5 — CLI live fetch mode and documentation

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Expose live fetching through the existing scenario CLI as an explicit opt-in, keeping fixtures the default, and document the new live capabilities end to end.

### Problem

Phases 1–4 add real-world acquisition capability, but the CLI still only runs fixture-backed scenarios. Operators need a clear, safe way to point a scenario at the live web without changing default deterministic behavior.

### Deliverables

- A CLI `--live` flag (and/or a scenario `fetchMode: "http"` field) that swaps `FixturePageFetcher` for `Psr18PageFetcher` for the **page fetch only**. Search evidence remains supplied.
- CLI surfacing of `FetchPolicy` options (timeout, max redirects, max body size, User-Agent, host allowlist/denylist) with safe defaults.
- Optional inclusion of robots.txt and sitemap evidence in live runs.
- A clear failure mode: if `--live` is requested but no PSR-18 client is wired/installed, the CLI exits non-zero with a clear stderr message explaining how to provide a client.
- README documentation: how to install a concrete PSR-18 client, how to run a live scenario, the safety defaults, and the determinism guarantee for the default (non-live) path.
- An example scenario (e.g. `examples/scenarios/live-fetch.json`) documented as live-only and excluded from deterministic test expectations.

### Acceptance criteria

- Default CLI behavior is unchanged: existing fixture scenarios run deterministically with no network.
- `--live` (or `fetchMode: "http"`) uses the HTTP fetcher for the page only; search evidence is still supplied.
- Requesting live mode without an available PSR-18 client fails fast with a clear stderr message and non-zero exit code.
- FetchPolicy options are configurable from the CLI with documented safe defaults.
- Tests cover flag parsing, fixture-vs-live fetcher selection (via injected mock client), and the missing-client error path — all without real network calls.
- README documents live usage, safety, and the deterministic default.
- `vendor/bin/phpunit` passes after implementation.

## v0.4 completion definition

v0.4 is complete when:

- the expected product page can be fetched over real HTTP through `Psr18PageFetcher`, behind the existing `PageFetcher` interface;
- the core depends only on PSR HTTP abstractions, not any concrete client;
- live fetching is bounded and gated by a deterministic `FetchPolicy` with safe defaults;
- `robots.txt` disallow of the expected URL is detected;
- sitemap presence/absence of the expected URL is detected;
- the CLI can run a scenario against the live web as an explicit opt-in, with fixtures remaining the default;
- the entire test suite remains deterministic and network-free using mock PSR-18 clients;
- existing JSON report compatibility and v0.2/v0.3 semantics are preserved;
- search-result acquisition remains out of the core.

## After v0.4

Possible v0.5+ directions are tracked in `docs/deferred.md`. The most likely next theme is **live search-result acquisition** (a real `SearchProvider` HTTP adapter), now that the HTTP plumbing, safety policy, and live CLI mode exist to build on.
