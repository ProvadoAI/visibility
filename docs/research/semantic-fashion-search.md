# semantic-fashion-search

## What this project actually does

`semantic-fashion-search` is now inspectable as a multi-component demo of semantic/hybrid product search for a Medusa fashion storefront. The checked-in source contains a Python FastAPI search service, a Medusa search plugin shim, a Medusa store configured to use that plugin, a Next.js storefront wired to the custom search endpoint, and a mostly stock Medusa admin application (`research/semantic-fashion-search/hybrid-search-engine-api/main.py`, `research/semantic-fashion-search/hybrid-search-engine-api/Search.py`, `research/semantic-fashion-search/medusa-plugin-customsearch/src/services/meilisearch.js`, `research/semantic-fashion-search/medusa-store/medusa-config.js`, `research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`, `research/semantic-fashion-search/medusa-admin/src/domain/products/components/general-form/index.tsx`).

The core search behavior lives in `hybrid-search-engine-api`. A `Search` object loads product metadata from Firestore, downloads a FAISS index and index mapping from Google Cloud Storage, loads CLIP text-embedding support, optionally loads a fine-tuned CLIP checkpoint, translates non-English queries to English through Google Translate with a glossary, retrieves product IDs from both vector search and Google Cloud Retail keyword search, fuses those ranked lists with reciprocal rank fusion, and returns product hit DTOs containing title, SKU list, handle, and thumbnail (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`).

The Medusa integration is pragmatic rather than complete. The custom search plugin subclasses Medusa's `SearchService`, leaves MeiliSearch indexing methods as no-ops, and forwards storefront product search requests to the custom backend API (`research/semantic-fashion-search/medusa-plugin-customsearch/src/services/meilisearch.js`). The Medusa store config still describes MeiliSearch-style searchable/displayed product attributes (`title`, `description`, `variant_sku`, `thumbnail`, and `handle`), but the plugin implementation routes searches to the semantic API instead of actually using MeiliSearch (`research/semantic-fashion-search/medusa-store/medusa-config.js`).

The storefront implements an InstantSearch-compatible client that reads the selected country/region, maps it to a language code, posts the query through Medusa's `/store/products/search` endpoint, passes API endpoint/key/top-N options to the plugin, and reshapes the response into `react-instantsearch-hooks-web` results (`research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`, `research/semantic-fashion-search/medusa-storefront/src/modules/search/templates/desktop-search-modal/index.tsx`, `research/semantic-fashion-search/medusa-storefront/src/modules/mobile-menu/components/search-menu/index.tsx`). Search result rendering is simple product-card output; it does not show match explanations, contributing fields, vector distance, keyword score, or diagnostic reasons (`research/semantic-fashion-search/medusa-storefront/src/modules/search/components/hit/index.tsx`, `research/semantic-fashion-search/medusa-storefront/src/modules/search/components/desktop-hits/index.tsx`).

## Useful concepts for our visibility-detection project

- **Semantic product-query matching.** The strongest concept is the separation between user query text and product retrieval through a product embedding index. `Search.get_text_emb()` tokenizes a query with CLIP, encodes it as normalized text features, and `__query_with_vector_search()` searches a FAISS index for nearest product IDs (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). For our project, this is useful later as an optional `ProductQueryMatcher` or `QueryExpectationProvider` inspiration: score whether a product plausibly matches an expected query. It should not be part of v0.1 core.
- **Hybrid search.** The API retrieves candidates from vector search and keyword search, then combines them instead of treating semantic search as a replacement for lexical search (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). This supports a future expected-query layer where semantic similarity, keyword overlap, SKU/ID exact matching, and catalog attributes can each contribute evidence.
- **Keyword search and query expansion.** Keyword retrieval uses Google Cloud Retail Search with `query_expansion_spec.condition = AUTO`, page size equal to the retrieval window, and product IDs returned from the first result page (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). This is a useful pattern for adapters that call provider search APIs, but not reusable in core because it depends on Google Cloud Retail.
- **Ranking/scoring by reciprocal rank fusion.** The implementation uses reciprocal rank fusion with a denominator constant of 60 and ranks from the vector and keyword lists (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). This is relevant to future query-product scoring and diagnostic explanations because it can preserve which retrieval channel contributed to a result. The current code discards individual channel scores in the response, so our design should keep evidence if we ever add ranking support.
- **Exact product identifier boost.** If the query exactly equals a product ID in the vector index mapping, the API forces that product ID to the top of the results (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). For visibility detection, the analogous idea is to distinguish query classes: exact SKU/GTIN/product-ID queries should have different expectations and confidence than broad category/descriptive queries.
- **Product indexing and catalog evidence.** Product metadata is loaded only for published products and includes title, description, handle, thumbnail, and gender (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). The Medusa-side transform helper flattens variant SKU/title/UPC/EAN/MID/HS code/options plus type, collection, and tags (`research/semantic-fashion-search/medusa-plugin-customsearch/src/utils/transform-product.js`). This is useful as a product-evidence inventory for future content-mismatch and expected-query matching.
- **Medusa product/catalog integration.** The store registers a custom search plugin with `products` settings for searchable and displayed attributes, while seed data shows how product titles, descriptions, handles, images, options, variants, inventory, and prices are represented in a Medusa catalog (`research/semantic-fashion-search/medusa-store/medusa-config.js`, `research/semantic-fashion-search/medusa-store/data/seed.json`). This is useful later for adapter design, not for the framework-agnostic PHP core.
- **Storefront search integration.** The custom search client adapts the semantic API to InstantSearch's expected response shape and lets existing search UI components remain largely unchanged (`research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`, `research/semantic-fashion-search/medusa-storefront/src/modules/search/components/desktop-hits/index.tsx`). The adapter pattern is useful: keep retrieval service specifics behind an interface and present stable result DTOs to consumers.
- **Locale/language handling.** The storefront maps selected country codes to language codes and the backend translates non-English queries to English with a glossary before retrieval (`research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`, `research/semantic-fashion-search/hybrid-search-engine-api/Search.py`). For visibility detection, locale/market should remain explicit on `SearchQuery` and provider evidence. Translation can change query intent, so it should be reported as evidence/limitation if used.

## Reusable implementation ideas

- Preserve a seam for a future `ProductQueryMatcher` that can return structured evidence such as lexical overlap, semantic similarity, exact identifier match, catalog-field coverage, locale handling, and confidence. The semantic-fashion-search vector/keyword/RRF split is a good conceptual input, but it should be optional and outside v0.1 core (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`).
- If we later add query expectation scoring, keep the individual retrieval signals instead of returning only fused product IDs. Evidence should include vector rank/distance, keyword rank, exact-ID match, fusion score, product fields used, and any translation applied. The inspected API currently prints timing and returns only product hits, so it is not evidence-rich enough for our reports (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`, `research/semantic-fashion-search/hybrid-search-engine-api/data.py`).
- Model exact identifier queries separately from broad product/category queries. The exact-ID boost in this project shows that SKU/product-ID matching should not be blended blindly with semantic/category matching (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`).
- Use adapter boundaries for ecommerce integrations. The Medusa plugin/storefront approach demonstrates that catalog/search plumbing can live outside the core while the retrieval service returns stable product hit DTOs (`research/semantic-fashion-search/medusa-plugin-customsearch/src/services/meilisearch.js`, `research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`).
- Treat product catalog fields as explicit evidence. Future detectors can compare expected query terms against title, description, variants, SKU/GTIN/UPC/EAN-like identifiers, collection, type, tags, handle, and availability/price fields where an adapter supplies them (`research/semantic-fashion-search/medusa-plugin-customsearch/src/utils/transform-product.js`, `research/semantic-fashion-search/medusa-store/data/seed.json`).

## What NOT to reuse

- Do not add FAISS, CLIP, Google Cloud Retail, Firestore, Google Translate, Google Cloud Storage, FastAPI, Medusa, React InstantSearch, or MeiliSearch dependencies to v0.1 core (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`, `research/semantic-fashion-search/medusa-plugin-customsearch/package.json`, `research/semantic-fashion-search/medusa-storefront/package.json`).
- Do not copy the custom search plugin as a production integration pattern. Its create/add/replace/delete indexing methods return `null`, so it is a search forwarding shim rather than a complete index lifecycle implementation (`research/semantic-fashion-search/medusa-plugin-customsearch/src/services/meilisearch.js`).
- Do not expose API keys or endpoint URLs in frontend code. The storefront hard-codes a search endpoint and API key with a comment noting it is not best practice (`research/semantic-fashion-search/medusa-storefront/src/lib/search-client.ts`).
- Do not use this project as a model for visibility diagnostics or merchant-facing explanations. Search results contain product fields, but no reason codes, no matched attributes, no score breakdown, no indexability evidence, no external SERP evidence, and no recommendations (`research/semantic-fashion-search/hybrid-search-engine-api/data.py`, `research/semantic-fashion-search/medusa-storefront/src/modules/search/components/hit/index.tsx`).
- Do not treat it as an external search visibility detector. It searches a merchant catalog, not Google/Bing/marketplace SERPs, and it does not test whether a known product URL appears in external results.
- Do not add semantic matching to the deterministic v0.1 pipeline. The current v0.1 priority remains static search-provider fixtures, URL normalization/matching, and explicit reports.

## Possible role in our future system

- **semantic product-query matching reference** — medium usefulness for future query expectation, especially CLIP text embeddings, FAISS nearest-neighbor retrieval, translation-to-English, exact-ID boosting, and separating product hit DTOs from storefront UI (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`, `research/semantic-fashion-search/hybrid-search-engine-api/main.py`).
- **hybrid search/ranking reference** — medium usefulness for combining vector and keyword result lists with reciprocal rank fusion, but the current response loses score/channel evidence that our reports would need (`research/semantic-fashion-search/hybrid-search-engine-api/Search.py`).
- **Medusa adapter reference** — low-to-medium usefulness for future ecommerce adapter boundaries and product-field mapping; the plugin is incomplete for index lifecycle and the admin app has no visibility-diagnostic workflow (`research/semantic-fashion-search/medusa-plugin-customsearch/src/services/meilisearch.js`, `research/semantic-fashion-search/medusa-store/medusa-config.js`, `research/semantic-fashion-search/medusa-admin/src/services/api.js`).
- **visibility diagnostics reference** — low usefulness. The project retrieves catalog products for a storefront search box; it does not explain why a product is absent from external search results, inspect product page indexability/canonicals/structured data, or generate findings/recommendations.

## Concrete lessons for our roadmap

- Keep `ProductQueryMatcher` / `QueryExpectationProvider` as a future seam, not a v0.1 implementation.
- If semantic matching is added later, return an evidence object rather than only a score: product ID, query, locale, translated query, semantic rank/distance, keyword rank, exact identifier match, fusion method, catalog fields used, and limitations.
- Treat lexical search and semantic/vector search as complementary signals. Hybrid matching is more useful for product-query expectation than embeddings alone.
- Keep locale/market/query-language fields explicit in `SearchQuery`; if translation is used by an adapter, surface it as evidence and possibly lower confidence.
- Distinguish exact identifier visibility expectations from descriptive/category-query expectations.
- Use ecommerce adapters to supply catalog/product evidence, but do not couple the PHP core to Medusa, Laravel, databases, queues, storefront UI, or search backend infrastructure.
- Keep v0.1 focused on deterministic external result evidence and URL matching. Semantic embeddings/vector infrastructure remains a later optional layer for deciding which queries should plausibly map to a product.
